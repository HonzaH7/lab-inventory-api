# Lab Inventory API

REST API for managing a laboratory reagent inventory тАФ built with Java 21, Spring Boot 3.5 and
PostgreSQL.

## About

Laboratories track chemical reagents by name, category, quantity, unit and expiration date. This API
provides the backend for that inventory: full CRUD over reagents, input validation, and consistent
error responses.

The domain was chosen deliberately тАФ I have a background in chemistry, so the model reflects how
reagents are actually catalogued rather than an arbitrary example domain.

Reagents are persisted in PostgreSQL through Spring Data JPA. The project began with an in-memory
store and was migrated to a real database; the [Design decisions](#design-decisions) section describes
what that migration cost.

## Tech stack

- **Java 21**, **Spring Boot 3.5**

- **PostgreSQL 16**, **Spring Data JPA** / Hibernate 6

## Getting started

### Prerequisites

- JDK 21 or newer
- Docker (for the database)

### 1. Start the database

```bash
docker run -d --name lab-inventory-db \
  -e POSTGRES_USER=labadmin \
  -e POSTGRES_PASSWORD=choose_a_password \
  -e POSTGRES_DB=lab_inventory \
  -p 5432:5432 \
  -v lab_inventory_data:/var/lib/postgresql/data \
  postgres:16
```

The named volume keeps the data across container restarts. On later runs the container already
exists, so starting it is enough:

```bash
docker start lab-inventory-db
```

> **Port 5432 must be free.** If a PostgreSQL instance is already installed natively on the machine it
> will hold that port, and the application will connect to it instead of the container тАФ producing a
> confusing `password authentication failed` error. Stop the native service, or publish the container
> on a different port and adjust `spring.datasource.url` accordingly.

### 2. Provide the database password

The password is never stored in the repository. Export it as an environment variable, using the same
value passed to `POSTGRES_PASSWORD` above:

```bash
export DB_PASSWORD=choose_a_password
```

On Windows PowerShell:

```bash
$env:DB_PASSWORD='choose_a_password'
```

In IntelliJ IDEA, add it under *Run тЖТ Edit Configurations тЖТ Modify options тЖТ Environment variables*.

### 3. Run

```bash
./mvnw spring-boot:run
```

The application starts on `http://localhost:8080`. On first startup Hibernate creates the `reagent`
table from the entity mapping, so no manual schema setup is required.

To see the generated SQL while developing, activate the `dev` profile:

```bash
./mvnw spring-boot:run -Dspring-boot.run.profiles=dev
```

### Run the tests

```bash
./mvnw test
```

Note that the tests currently require a running database тАФ `LabInventoryApiApplicationTests` boots the
full application context, which includes the datasource. Removing that dependency is on the
[Roadmap](#roadmap).

## Configuration

| Property | Environment variable | Default | Description |
|----------|---------------------|---------|-------------|
| `spring.datasource.url` | тАФ | `jdbc:postgresql://localhost:5432/lab_inventory` | Database URL |
| `spring.datasource.username` | тАФ | `labadmin` | Database user |
| `spring.datasource.password` | `DB_PASSWORD` | none тАФ **required** | Database password |
| `spring.jpa.hibernate.ddl-auto` | тАФ | `update` | Schema management strategy |

The password is deliberately declared as `${DB_PASSWORD}` without a fallback value. A missing password
makes the application fail at startup rather than silently attempting to connect with a default.

Profiles live in `src/main/resources`:

- `application.yml` тАФ base configuration
- `application-dev.yml` тАФ development profile, logs the SQL statements Hibernate generates

## API

### Endpoints

| Method | Path | Description | Success | Errors |
|--------|------|-------------|---------|--------|
| `GET` | `/reagents` | List all reagents | `200` | тАФ |
| `GET` | `/reagents/{id}` | Get one reagent | `200` | `404` |
| `POST` | `/reagents` | Create a reagent | `201` | `400` |
| `PUT` | `/reagents/{id}` | Replace a reagent | `200` | `400`, `404` |
| `DELETE` | `/reagents/{id}` | Delete a reagent | `204` | `404` |

### Reagent fields

| Field | Type | Constraints |
|-------|------|-------------|
| `id` | `Long` | generated by the database, never sent by the client |
| `name` | `String` | required, not blank |
| `category` | enum | required тАФ `ACID`, `BASE`, `SALT` |
| `amount` | `double` | required, must not be negative |
| `unit` | enum | required тАФ `ML`, `L`, `G`, `KG`, `MG`, `MOL` |
| `expiration` | `LocalDate` | optional, ISO format (`yyyy-MM-dd`) |

### Example тАФ create a reagent

```http
POST /reagents
Content-Type: application/json

{
  "name": "Hydrochloric acid",
  "category": "ACID",
  "amount": 100,
  "unit": "ML",
  "expiration": "2027-06-01"
}
```

```http
201 Created

{
  "id": 1,
  "name": "Hydrochloric acid",
  "category": "ACID",
  "amount": 100.0,
  "unit": "ML",
  "expiration": "2027-06-01"
}
```

### Example тАФ validation failure

```http
POST /reagents
Content-Type: application/json

{ "name": "", "category": "ACID", "amount": -5, "unit": "ML" }
```

```http
400 Bad Request

{
  "name": "name is required",
  "amount": "amount cannot be negative"
}
```

### Example тАФ resource not found

```http
GET /reagents/999
```

```http

{
  "message": "Reagent with id 999 not found"
}
```

## Architecture

```
cz.jan.labinventoryapi
тФЬтФАтФА controller    HTTP layer тАФ request mapping, status codes, DTO mapping
тФЬтФАтФА service       business logic тАФ existence rules, transaction boundaries
тФЬтФАтФА repository    data access тАФ Spring Data JPA repository
тФЬтФАтФА model         domain model / JPA entities (Reagent, Category, Unit)
тФЬтФАтФА dto           API contract (ReagentRequest, ReagentResponse)
тФФтФАтФА exception     custom exceptions + global handler
```

### Design decisions

**Separate DTOs from the domain model.** `ReagentRequest` (input) has no `id` тАФ the database assigns
it, the client never sends it. `ReagentResponse` (output) does. Keeping these separate from `Reagent`
means the persistence model can evolve without breaking the public API contract, and internal fields
are never exposed by accident.

**Constructor injection everywhere.** Dependencies are `final` and visible in the constructor
signature, which keeps components immutable, makes dependencies explicit, and allows instantiating
them in tests without a Spring context.

**Centralised error handling.** A single `@RestControllerAdvice` translates exceptions into HTTP
responses тАФ validation failures into `400` with per-field messages, `ReagentNotFoundException` into
`404`. Controllers contain no error-handling code, and every endpoint returns errors in the same shape.

**Business rules live in the service layer.** Rules such as "a reagent must exist before it can be
updated" belong neither to the HTTP layer nor to storage. Keeping them in the service means they hold
regardless of how the logic is invoked, and they survived replacing the storage implementation.

**Enums are persisted as strings.** `category` and `unit` are mapped with `EnumType.STRING` rather
than the default ordinal mapping. Ordinals store the position of a constant, so inserting a new value
into the middle of an enum would silently redefine every existing row. As a side effect Hibernate
generates `CHECK` constraints from the enum values, so invalid categories are rejected by the database
itself and not only by the application.

**What the storage abstraction actually bought.** The original design hid the in-memory store behind a
generic `Repository<ID, T>` interface, on the assumption that moving to a database would leave the
controller and service layers untouched. Half of that held: the controller did not change by a single
line. The service did. Spring Data's `save(entity)` replaced the hand-written `save(id, entity)`, id
generation moved from an `AtomicLong` to the database, and `update` had to be rewritten to mutate a
loaded entity instead of constructing a new one тАФ the previous version would have inserted a duplicate
row rather than updating the existing one. The abstraction was worth having, but it protected the
layer above it rather than the one using it directly.

## Schema management

Hibernate currently generates the schema from the entity mapping (`ddl-auto: update`). This is
convenient during development but is not a production strategy: it only adds tables and columns, never
alters or removes them, and the resulting schema is a side effect of the code rather than something
reviewable. Migrating to Flyway with `ddl-auto: validate` is on the roadmap.

## Testing

- `@WebMvcTest` slice tests for the controller тАФ the service is mocked, so the tests verify HTTP
  concerns only: routing, status codes, JSON structure, and delegation to the service.
- Error paths are covered as well: a missing reagent is asserted to produce `404` through the global
  exception handler.
- Because `id` is now assigned by the database, the controller tests set it with
  `ReflectionTestUtils` rather than through a constructor тАФ the entity intentionally exposes no
  setter for it.

```bash
./mvnw test
```
